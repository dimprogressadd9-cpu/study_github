## このリポジトリの目的

プロジェクトリーダーになる準備として、**開発管理プラットフォームとしての GitHub / GitLab**
を実習で学ぶための練習用リポジトリです。コードの正しさよりも、
**プラットフォームの使い方(タスク管理・コードレビュー・CI/CD)を体で覚えること**が目的です。

詳細な学習計画は `github_gitlab_leader_roadmap.md` にあります。
具体的な手順や YAML サンプルが必要なときは、まずそのファイルを参照してください。

---

## 学習者について

- Git の基本(コミット・ブランチ・リポジトリ操作)は GitBlit + SourceTree で習得済み。
- これから学びたいのは、その上位にある「プラットフォーム機能」:
  1. Issue / ボードによるタスク・スケジュール管理
  2. Merge Request / Pull Request とコードレビュー運用
  3. CI/CD による自動化(ビルド・テスト・デプロイ)
- 仕事では **GitLab**(社内規定)、個人 / Claude 学習では **GitHub** を使う。
- チーム規模は少人数(〜5名)を想定。
- 進め方の好み: **手を動かす実習中心**。

## Claude Code への依頼スタンス(重要)

ここは「学習用」リポジトリです。先回りして全部完成させないでください。

- **チューターとして振る舞う**。作業を肩代わりするのではなく、
  学習者が自分で操作できるよう手順を示し、必要な箇所だけ補助する。
- ファイルや設定を作るときは、**何を・なぜそうするのかを短く説明**してから進める。
- まとめて大量に生成せず、**1ステップずつ**進め、学習者が確認・操作する余地を残す。
- 「ここは自分でやってみますか?」と区切りを入れてよい。
- GitHub と GitLab の対応関係(PR↔MR、Actions↔CI/CD など)は折に触れて補足する。

## このリポジトリの規約(決めたら更新していく)

学習しながら確立していく約束事。最初は仮置きで、慣れたら調整する。

- **ブランチ戦略**: GitHub Flow。`main` + 短命の `feature/*` ブランチ + PR/MR。
- **ブランチ命名**: `feature/...`, `fix/...`, `docs/...`。
- **コミットメッセージ**: 1行目は要約(例: `feat: add input validation`)。
  慣れてきたら Conventional Commits(`feat` / `fix` / `docs` / `chore` 等)に寄せる。
- **マージ方式**: Squash and merge(履歴を簡潔に保つ)。
- **main は保護**: 直接 push 禁止。PR/MR + CI 成功を必須にする(フェーズ2以降で設定)。
- **作業はすべて Issue から始める**: PR/MR には `Closes #<番号>` で Issue を紐づける。
- **ラベル命名**: 接頭辞で体系化。`type:bug` / `type:feature` /
  `priority:high|med|low` / `status:todo|in-progress|review|done`。

## 環境メモ(忘れがちな注意点)

- **GitHub の学習用リポジトリは public にする**: public なら標準ランナーの
  GitHub Actions が無制限・無料。private は月の無料枠を消費する。
- **GitLab.com の無料枠は CI/CD が月400分と少ない**: パイプラインは軽量に保ち、
  `cache:` を活用、不要ジョブは `rules:` でスキップする。
- **GitLab の上位計画機能(エピック / ロードマップ / 詳細な承認ルール)は有料(Premium)**:
  無料枠では「マイルストーン + スコープ付きラベル + パイプライン必須 + 全コメント解決必須」
  で品質を担保する設計にする。
